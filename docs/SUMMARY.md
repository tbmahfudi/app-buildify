## What is App-Buildify?

App-Buildify is a **multi-tenant NoCode/LowCode platform** for building business applications without custom code. It provides:

- Multi-tenancy with isolated data per tenant
- Fine-grained Role-Based Access Control (RBAC)
- Visual data model designer (NoCode entities)
- Workflow engine and automation rules
- Dashboard builder and report designer
- Pluggable module system
- Multi-language support (EN, DE, ES, FR, ID)
- Full audit logging

**Technology stack**: FastAPI (Python 3.11) | PostgreSQL 15 | SQLAlchemy 2.0 | Redis 7 | Vanilla JS (ES6+) | Tailwind CSS | Docker

## Platform Overview

### System Architecture

```
┌─────────────────────────── API Gateway (Nginx) ────────────────────────────┐
│          Routes: /api → Core Platform | /api/v1/financial → Module          │
└───────────────────────┬────────────────────────┬───────────────────────────┘
                        │                        │
           ┌────────────▼──────────┐   ┌─────────▼──────────┐
           │  Core Platform API    │   │  Financial Module  │
           │  FastAPI :8000        │   │  FastAPI :9001     │
           └────────────┬──────────┘   └─────────┬──────────┘
                        └──────────────┬──────────┘
                              PostgreSQL + Redis
                           Frontend (Vanilla JS / Nginx)
```

### Multi-Tenancy Model

```
Platform → Tenant → Company → Branch → Department
```

- A User belongs to one Tenant and can access one or more Companies
- All DB queries are automatically filtered by `tenant_id` and `company_id`

### Module System

1. Module registers via `POST /api/v1/modules/register`
2. Tenant enables the module in the admin UI
3. Module menus, routes, and permissions are dynamically injected
4. Each module ships with its own FastAPI service, DB migrations, and frontend pages

## API Reference

All endpoints: `Authorization: Bearer <token>` header required (except `/auth/login`).

### Auth

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/auth/login` | Authenticate → access + refresh tokens |
| `POST` | `/auth/refresh` | Exchange refresh token for new access token |
| `POST` | `/auth/logout` | Revoke current access token |
| `POST` | `/auth/forgot-password` | Initiate password reset |
| `POST` | `/auth/reset-password` | Complete password reset |

### Key Resource Endpoints (pattern)

All major resources follow standard REST CRUD:
- `GET /resource` — list (with `?page=1&page_size=20&sort_by=X&sort_order=desc`)
- `POST /resource` — create
- `GET /resource/{id}` — read one
- `PUT /resource/{id}` — update
- `DELETE /resource/{id}` — delete

**Resources**: `/users`, `/rbac/roles`, `/rbac/permissions`, `/rbac/groups`, `/org/tenants`, `/org/companies`, `/org/branches`, `/org/departments`, `/data-model/entities`, `/dynamic-data/{entity_name}`, `/workflows`, `/automations`, `/dashboards`, `/reports`, `/lookups`, `/scheduler/jobs`, `/audit`, `/modules`

### Response Envelope (lists)

```json
{ "items": [...], "total": 100, "page": 1, "page_size": 20, "pages": 5 }
```

### HTTP Status Codes

`200` OK | `201` Created | `400` Bad request | `401` Unauthenticated | `403` Forbidden | `404` Not found | `409` Conflict | `422` Schema error | `423` Account locked | `429` Rate limited | `500` Server error

## Authentication & Security

### JWT Flow

1. `POST /auth/login` → bcrypt password check → issue access token (30 min) + refresh token (7 days)
2. Every request: decode JWT → check Redis blacklist → load user → RBAC check
3. `POST /auth/logout` → add token `jti` to Redis blacklist (TTL = remaining expiry)
4. Auto-refresh: 401 → client retries with refresh token → new access token

### Password Policy (configurable per tenant)

| Setting | Default |
|---------|---------|
| Min length | 8 |
| Require uppercase/lowercase/digit/special | true |
| History count | 5 (cannot reuse last N) |
| Max age | 90 days |
| Expiry warning | 7 days before |

### Account Lockout

- Default: 5 failed attempts → 15 min lockout (progressive: doubles each time)
- Configurable: `fixed` or `progressive` lockout types

### Session Management

- Idle timeout: 30 min | Absolute timeout: 8 h | Max concurrent sessions: 3
- `single_session_mode` available

### Rate Limiting

- Default: 100 req/min per IP (via SlowAPI)
- Login: 10 req/min

### Security Headers

`Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `X-XSS-Protection`, `Strict-Transport-Security`, `Referrer-Policy`

### Audit Events

`auth.login.success/failed`, `auth.logout`, `auth.password.changed/reset`, `user.lockout/unlocked`, `rbac.role.assigned`, `data.*` (all data modifications)

## RBAC System

### Permission Format

```
resource:action:scope
```

Examples: `users:create:tenant` | `data:*:branch` | `financial:invoices:read:company` | `*:*:platform`

### Data Model

```
User → UserRole → Role → RolePermission → Permission
User → UserGroup → Group → GroupRole → Role → Permission
```

Effective permissions = direct role permissions + group role permissions

### Default Roles (seeded)

| Role | Scope | Access |
|------|-------|--------|
| `SuperAdmin` | Platform | `*:*:platform` — full access |
| `TenantAdmin` | Tenant | All tenant operations |
| `CompanyAdmin` | Company | All company operations |
| `Manager` | Company | Read/write most resources |
| `User` | Company | Read-only most resources |

### Adding a New Permission

1. Define: `resource:action:scope`
2. Create: `POST /api/v1/rbac/permissions`
3. Assign to roles
4. Guard API: `Depends(require_permission("..."))`
5. Filter UI: `hasPermission('...')`

### Wildcard Support

`users:*:tenant` (any action) | `data:read:*` (any scope) | `*:*:platform` (full access)

## Frontend Architecture

**Stack**: Vanilla JS (ES6+) | Flex Component Library (zero dependencies) | Tailwind CSS (CDN) | Phosphor Icons | i18next | Vitest

### Bootstrap Flow

```
index.html → app-entry.js → i18next init → load main.html
  → app.js → check tokens → load user + permissions
  → merge core menu + module menus → RBAC filter → init router → navigate
```

### Routing (hash-based)

`#/dashboard` | `#/users` | `#/rbac` | `#/tenants` | `#/companies` | `#/nocode/data-model` | `#/nocode/workflows` | `#/nocode/automations` | `#/menu` | `#/settings` | `#/modules` | `#/financial/*` (dynamic)

### API Client (`api.js`)

Tokens stored in `localStorage`. Methods: `apiGet/Post/Put/Delete()`, `login()`, `logout()`, `refreshToken()`. Auto-refresh on 401.

### State

Simple module-level `appState = { user, currentRoute }`. Access via `getAppState()` / `getCurrentUser()`.

### Menu System

Core menu (`frontend/config/menu.json`) merged with enabled module manifests at runtime, then RBAC-filtered.

### Testing

```bash
npm test                # Vitest unit tests
npm run coverage        # Coverage report (targets: 80% statements/functions/lines, 75% branches)
```

**Layout** (9): `FlexStack`, `FlexGrid`, `FlexContainer`, `FlexSection`, `FlexSidebar`, `FlexCluster`, `FlexToolbar`, `FlexMasonry`, `FlexSplitPane`

**UI** (16): `FlexCard`, `FlexModal`, `FlexTabs`, `FlexDataGrid`, `FlexTable`, `FlexBadge`, `FlexStepper`, `FlexAlert`, `FlexButton`, `FlexDrawer`, `FlexDropdown`, `FlexSpinner`, `FlexAccordion`, `FlexBreadcrumb`, `FlexTooltip`, `FlexPagination`

**Form** (5): `FlexInput`, `FlexSelect`, `FlexCheckbox`, `FlexRadio`, `FlexTextarea`

**Bundle**: Core ~12 KB gzipped | Full ~38 KB gzipped

### Planned (Not Yet Implemented)

`FlexDatepicker` (High), `FlexFileUpload` (High), `FlexForm` (High), `FlexNotification` (Medium), `FlexProgress` (Medium)

### Adding a Component

1. Create `frontend/assets/js/components/flex-my-widget.js`
2. Extend `BaseComponent`, implement `render()`
3. Register with `customElements.define('flex-my-widget', FlexMyWidget)`
4. Export and register in `components/index.js`

### Accessibility

WCAG 2.1 AA target: semantic HTML, ARIA roles, full keyboard nav, focus traps in modals/drawers, color contrast ≥ 4.5:1.

## Internationalization

**Library**: i18next + i18next-http-backend (lazy loading)

**Supported languages**: `en` (default), `de`, `es`, `fr`, `id`

### Namespaces

| File | Content |
|------|---------|
| `common.json` | Global reusable terms (Save, Cancel, Loading...) |
| `pages.json` | Page-specific labels with dot-notation keys |
| `menu.json` | Navigation labels |

### Usage

```javascript
t('save')                                    // "Save" (common namespace)
t('pages:dashboard.title')                   // "Dashboard"
t('pages:dashboard.welcome', { name: 'Jane' }) // "Welcome, Jane!"
```

### Language Switching

```javascript
localStorage.setItem('language', lang)
i18next.changeLanguage(lang)
```

### Adding a Language

1. Create `frontend/assets/i18n/<code>/` with `common.json`, `pages.json`, `menu.json`
2. Add to `supportedLngs` in `i18n.js`
3. Add option to settings dropdown in `settings.js`

## Deployment

### Docker Services

| Service | Port | Description |
|---------|------|-------------|
| `postgres` | 5432 | PostgreSQL 15 |
| `backend` | 8000 | Core FastAPI platform |
| `financial-module` | 9001 | Financial module API |
| `frontend` | 8080 | Nginx SPA + API proxy |

### Nginx Routing

| Pattern | Destination |
|---------|------------|
| `/api/v1/financial/*` | `financial-module:9001` |
| `/api/` | `backend:8000` |
| `/modules/<name>/<file>` | Module frontend assets |
| `/*` (fallback) | `index.html` (SPA) |

### Migrations

```bash
docker-compose exec backend alembic upgrade head    # Apply
docker-compose exec backend alembic downgrade -1   # Roll back one
docker-compose exec backend alembic current        # Check state
```

### Adding a New Module

1. Create `modules/<name>/` directory
2. Add service to `infra/docker-compose.dev.yml`
3. Add Nginx route pattern to `infra/nginx/app.conf`
4. Register: `POST /api/v1/modules/register`

### Backup

```bash
docker-compose exec postgres pg_dump -U appuser appdb > backup_$(date +%Y%m%d).sql
docker-compose exec -T postgres psql -U appuser appdb < backup.sql
```

## Production Checklist

### Critical security items before going live

- [ ] `JWT_SECRET_KEY` is a strong 64-char random secret
- [ ] Database and Redis passwords are strong and unique
- [ ] Default admin password changed
- [ ] `DEBUG=false`, `ENVIRONMENT=production`
- [ ] `ALLOWED_ORIGINS` restricted to production domain
- [ ] TLS/HTTPS enabled at Nginx
- [ ] `.env` files not committed to git
- [ ] `?sslmode=require` appended to `DATABASE_URL`
- [ ] Sentry DSN configured

### Generate secure secrets

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(48))"   # SECRET_KEY
python3 -c "import secrets; print(secrets.token_hex(16))"        # DB/Redis password
```

### Smoke tests after deployment

1. `GET /health` → 200 on all services
2. Admin login succeeds
3. `GET /api/v1/users` returns data with valid token
4. Frontend loads without console errors
5. Data model created and record saved
6. Audit log shows the operations above

## Financial Module

**Status**: Production Ready (v1.0.0) | **Port**: 9001 | **API prefix**: `/api/v1/financial/`

### Features

Double-entry bookkeeping, Chart of Accounts (hierarchical, 50-account default template), Customer management, Invoicing with status workflow (`DRAFT → POSTED → PARTIALLY_PAID → PAID / VOID`), Payment recording and allocation, Journal entries with reversal, Tax rate management, and 6 financial reports.

### Reports

`/reports/trial-balance` | `/reports/balance-sheet` | `/reports/income-statement` | `/reports/aged-receivables` | `/reports/cash-flow` | `/reports/account-ledger/{id}`

### API Summary

| Router | Endpoints | Key operations |
|--------|-----------|---------------|
| `/accounts` | 10 | CRUD, hierarchy, balance, default chart setup |
| `/customers` | 8 | CRUD, balance, invoices, overdue list |
| `/invoices` | 7 | CRUD, post, void |
| `/payments` | 9 | CRUD, allocate, clear, void, unallocated list |
| `/journal-entries` | 7 | CRUD, post, reverse |
| `/tax-rates` | 7 | CRUD, active list, calculate |
| `/reports` | 6 | Trial balance, balance sheet, P&L, aged receivables, cash flow, ledger |

### Permissions

Format: `financial:<resource>:<action>:<scope>`

Key permissions: `financial:accounts:read:company`, `financial:invoices:write:company`, `financial:invoices:post:company`, `financial:journal:post:company`, `financial:reports:read:company`, `financial:*:*:company` (full access)

### Event Bus Integration

**Publishes**: `financial.invoice.created/posted/paid`, `financial.payment.recorded`, `financial.journal.posted`
**Consumes**: `tenant.created` (init default chart), `company.created` (setup financial config)

### Setup

```bash
docker-compose exec financial-module alembic upgrade head
POST /api/v1/financial/accounts/setup/default
  { "tenant_id": "<uuid>", "template": "standard" }
```
